Which architecture for the accounting frontend?

Three approaches — click the one that feels right

A

SPA + Direct Supabase + Thin Import Server ⭐ Recommended

React SPA reads directly from Supabase (14 tables + 2 SQL report functions already tested). A minimal Hono server handles only SIE4 file import and Fortnox PDF parsing (requires Node.js for iconv-lite + cheerio).

Pros
  • Simplest reads — browser → Supabase directly
  • Report SQL functions callable via client.rpc()
  • Server is ~100 lines, reuses existing pipeline
  • Can add server endpoints later if needed
Cons
  • Complex queries (Huvudbok) awkward in supabase-js
  • Still need a server process for import
B

Full-Stack with Hono API

React frontend + Hono REST API for everything. API wraps all DB queries. Same pattern as AI-REDOVISNING. Frontend never touches Supabase directly.

Pros
  • Full control over every query
  • Server computes running balances, subtotals
  • Proven AI-REDOVISNING pattern
Cons
  • 2× boilerplate (route handler + client hook for every query)
  • Extra latency: browser → Hono → Supabase
  • More code to maintain
C

SPA + Supabase Edge Functions

Same as A for reads, but Edge Functions for import and validation. No separate server.

Pros
  • No server process to manage
  • Serverless scaling
Cons
  • iconv-lite Buffer compat issues in Deno
  • cheerio (~1.5MB) near 2MB Edge limit
  • Harder to debug locally
  • Cold start latency

Data flow comparison

Approach A (Recommended)
📂 SIE4 file → Import Server (Hono, 2 endpoints) → Supabase
📊 All reads → Browser → Supabase directly (supabase-js)
📋 Reports → Browser → client.rpc('report_balansrapport')
✅ Validation → Browser runs checks + Server parses Fortnox PDF
